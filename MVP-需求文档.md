# 基于 Next.js + Vercel AI SDK 的私人 AI 助手 MVP 需求文档

## 1. 产品目标

### 1.1 产品定位
构建一个面向个人用户的 AI 助手 Web 应用，基于 `Next.js` 和 `Vercel AI SDK` 实现，帮助用户以自然语言完成日常信息整理、问答检索、内容生成和简单任务辅助。

### 1.2 核心目标
- 提供一个可随时使用的对话式 AI 助手入口。
- 支持连续多轮对话，并保留基础上下文记忆。
- 支持常见个人场景下的内容生成与整理能力。
- 形成一个可持续迭代的产品底座，后续可以接入工具调用、知识库、自动化任务等增强能力。

### 1.3 MVP 成功标准
- 用户可以注册或登录后创建并使用自己的 AI 助手。
- 用户能够发起多轮对话，并查看历史会话。
- AI 可以稳定返回回答，支持流式输出。
- 用户能够在一个会话中获取常见任务支持，例如总结、改写、计划生成、问答。
- 系统具备最基础的数据持久化、权限隔离和错误处理能力。

## 2. 用户场景

### 2.1 目标用户
- 希望拥有一个“私人 Copilot”的个人用户。
- 需要日常写作、整理、总结、规划支持的知识工作者。
- 希望用低门槛方式体验 AI 助手能力的独立开发者或小团队成员。

### 2.2 典型使用场景

#### 场景一：日常问答助手
用户向助手提问，例如“帮我解释这段概念”或“帮我整理这份信息”，助手给出清晰答复，并支持连续追问。

#### 场景二：内容生成与改写
用户输入主题、草稿或要点，助手输出邮件、周报、文案、学习笔记、会议纪要等文本内容。

#### 场景三：个人计划整理
用户描述目标，例如“帮我制定一周学习计划”，助手根据目标、时间和约束生成可执行计划。

#### 场景四：会话沉淀与回看
用户之前与助手聊过一个主题，之后再次进入产品时，可以查看历史会话，延续上下文或重新参考结果。

## 3. MVP 范围

### 3.1 必做范围
- 基础用户系统
  - 支持邮箱或第三方登录。
  - 每个用户只能访问自己的数据。
- 对话能力
  - 创建新会话。
  - 发送消息并获得 AI 回复。
  - 支持流式返回。
  - 支持多轮上下文。
- 会话管理
  - 查看历史会话列表。
  - 进入某条历史会话继续对话。
  - 支持重命名会话标题。
- 基础助手能力
  - 通用问答。
  - 文本总结。
  - 文本改写。
  - 计划/清单生成。
- 持久化能力
  - 保存用户、会话、消息数据。
  - 保存会话更新时间，便于列表排序。
- 基础产品体验
  - 桌面端与移动端可用。
  - 有加载、错误、空状态处理。
  - 支持基础限流与异常兜底。

### 3.2 可选但不强依赖
- 预置提示词模板，例如“总结模式”“写作模式”“计划模式”。
- 简单的用户设置页，例如默认模型、昵称。
- 对 AI 回复进行复制、重新生成。

## 4. 非目标

以下内容不纳入第一版 MVP：
- 不做复杂多 Agent 协作。
- 不做企业级团队协作、成员管理、权限分层。
- 不做深度知识库检索和 RAG 系统。
- 不做复杂工具调用编排，例如日历、邮件、Notion、搜索、数据库写入。
- 不做长期记忆系统和高级画像推荐。
- 不做语音输入输出、视频或图像生成。
- 不做精细化计费、订阅和商业化后台。
- 不做复杂工作流自动化。

## 5. 功能模块

### 5.1 用户与身份模块
功能说明：
- 用户注册、登录、退出。
- 获取当前登录态。
- 将数据按用户隔离。

MVP 要求：
- 优先使用成熟认证方案，如 `NextAuth`、`Clerk` 或 `Auth.js`。
- 未登录用户不可访问个人会话数据。

### 5.2 会话模块
功能说明：
- 创建新会话。
- 查看会话列表。
- 编辑会话标题。
- 删除会话。

MVP 要求：
- 新会话可自动生成默认标题。
- 会话列表按最近更新时间倒序排列。
- 删除操作可做软删除或直接删除，MVP 可优先直接删除。

### 5.3 消息模块
功能说明：
- 用户发送消息。
- AI 返回消息。
- 在同一会话中保留消息顺序。

MVP 要求：
- 消息类型至少包含 `user` 和 `assistant`。
- 支持流式输出，提升交互体验。
- 消息发送失败时支持提示与重试。

### 5.4 AI 编排模块
功能说明：
- 接收前端请求。
- 组装上下文消息。
- 调用模型生成回答。
- 以流式方式返回前端。

MVP 要求：
- 基于 `Vercel AI SDK` 的聊天能力实现。
- 初期优先单模型方案，避免模型路由复杂度。
- 控制上下文长度，避免 token 失控。

### 5.5 助手能力模块
功能说明：
- 以通用聊天为主，辅以少量高频模式。

MVP 建议能力：
- 通用问答。
- 内容总结。
- 内容润色与改写。
- 清单、计划、结构化建议生成。

实现建议：
- 通过 system prompt + 用户意图引导实现，不额外做复杂工作流。
- 模式选择可以先通过简单快捷入口或预置 prompt 实现。

### 5.6 前端交互模块
功能说明：
- 聊天主界面。
- 会话侧边栏。
- 新建会话入口。
- 历史会话切换。

MVP 要求：
- 聊天区域支持流式渲染。
- 输入框支持多行输入。
- 支持“发送中”状态。
- 基础 Markdown 渲染可选接入，但不是硬性必需。

### 5.7 系统稳定性模块
功能说明：
- 错误日志。
- 请求限流。
- 基础监控。

MVP 要求：
- 至少记录接口错误和模型调用失败。
- 防止用户高频刷接口导致成本异常。

## 6. 数据模型

以下为建议的核心数据模型，MVP 可使用 `Postgres` 或 `SQLite` 起步。

### 6.1 User

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 用户唯一 ID |
| email | string | 邮箱，唯一 |
| name | string | 用户昵称 |
| avatarUrl | string | 头像地址，可空 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

### 6.2 Conversation

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 会话唯一 ID |
| userId | string | 所属用户 ID |
| title | string | 会话标题 |
| lastMessageAt | datetime | 最后一条消息时间 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

### 6.3 Message

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 消息唯一 ID |
| conversationId | string | 所属会话 ID |
| role | enum | `user` / `assistant` / `system` |
| content | text | 消息内容 |
| status | enum | `pending` / `success` / `error` |
| tokenUsage | int | token 消耗，可空 |
| model | string | 使用模型标识，可空 |
| createdAt | datetime | 创建时间 |

### 6.4 UserSetting

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 设置记录 ID |
| userId | string | 用户 ID |
| defaultModel | string | 默认模型，可空 |
| systemPrompt | text | 用户级默认助手设定，可空 |
| createdAt | datetime | 创建时间 |
| updatedAt | datetime | 更新时间 |

### 6.5 实体关系
- 一个 `User` 可以拥有多个 `Conversation`。
- 一个 `Conversation` 可以拥有多条 `Message`。
- 一个 `User` 最多对应一条 `UserSetting`，也可以按业务允许多条版本化设置。

## 7. 技术架构

### 7.1 总体架构
- 前端：`Next.js` App Router
- UI 层：React + Tailwind CSS + 组件库
- AI 接入层：`Vercel AI SDK`
- 后端接口：`Next.js Route Handlers` 或 `Server Actions`
- 数据库：`Postgres` 优先，MVP 阶段也可先用 `SQLite`
- ORM：`Prisma` 或 `Drizzle`
- 鉴权：`Auth.js / NextAuth / Clerk`
- 部署：`Vercel`

### 7.2 架构分层

#### 表现层
- 聊天页面
- 会话列表
- 登录/设置页面

职责：
- 渲染 UI
- 管理输入状态
- 处理流式响应展示

#### 应用层
- 会话创建与查询逻辑
- 消息发送逻辑
- 用户权限校验

职责：
- 编排前端请求和业务流程
- 决定何时读写数据库
- 组装模型输入

#### AI 服务层
- system prompt 管理
- 历史消息裁剪
- 模型调用
- 流式结果返回

职责：
- 与 `Vercel AI SDK` 对接
- 屏蔽底层模型差异

#### 数据层
- 用户数据
- 会话数据
- 消息数据

职责：
- 数据持久化
- 查询性能和索引支持

### 7.3 关键技术决策

#### 决策一：采用 Next.js App Router
原因：
- 与 Vercel 部署链路契合度高。
- 更适合统一前后端工程结构。
- 易于结合 Route Handlers、Server Components 和 Server Actions。

#### 决策二：采用 Vercel AI SDK 做流式聊天
原因：
- 能快速搭建聊天 MVP。
- 对流式输出、消息结构、UI 集成支持较成熟。
- 后续扩展工具调用、多模态和结构化输出更平滑。

#### 决策三：初期只接入单模型
原因：
- MVP 目标是验证产品价值，不是验证复杂模型路由。
- 单模型可以降低成本、监控和调试复杂度。

#### 决策四：会话与消息强持久化
原因：
- 历史回看是私人助手最基础的体验。
- 后续做记忆、搜索和分析都依赖结构化消息沉淀。

### 7.4 API 建议

#### `POST /api/chat`
用途：
- 发送用户消息并获得 AI 流式回复。

请求体示例：
- `conversationId`
- `message`
- `mode`

响应：
- 流式文本数据。

#### `GET /api/conversations`
用途：
- 获取当前用户的会话列表。

#### `POST /api/conversations`
用途：
- 创建新会话。

#### `PATCH /api/conversations/:id`
用途：
- 更新会话标题。

#### `DELETE /api/conversations/:id`
用途：
- 删除会话。

#### `GET /api/conversations/:id/messages`
用途：
- 获取某个会话的消息列表。

## 8. 开发顺序

### 阶段一：项目初始化
- 初始化 `Next.js` 项目。
- 集成 `Tailwind CSS`、基础 UI 组件。
- 配置 `Vercel AI SDK`。
- 选定数据库与 ORM。
- 接入鉴权方案。

输出结果：
- 项目可启动。
- 用户可登录。
- 基础页面框架可访问。

### 阶段二：核心聊天链路
- 完成聊天页面基础布局。
- 完成 `POST /api/chat`。
- 接入模型流式输出。
- 打通前端发送消息到后端返回结果。

输出结果：
- 用户可以在前端看到 AI 实时回复。

### 阶段三：会话持久化
- 建立 `Conversation` 与 `Message` 表。
- 支持新建会话、读取历史消息。
- 支持会话列表展示与切换。

输出结果：
- 对话内容在刷新后不会丢失。
- 用户可以继续历史会话。

### 阶段四：产品体验完善
- 自动生成会话标题。
- 增加加载态、错误态、空状态。
- 支持删除会话、重命名会话。
- 优化移动端布局。

输出结果：
- 产品达到可演示、可测试的 MVP 状态。

### 阶段五：基础运营与稳定性
- 增加日志监控。
- 增加限流与异常兜底。
- 记录基础使用数据，例如消息数、活跃会话数。

输出结果：
- 产品具备最基础的线上可用性。

## 9. MVP 验收标准

- 用户可以完成注册或登录，并进入聊天界面。
- 用户可以新建会话并发送消息。
- AI 回复支持流式展示，平均响应体验可接受。
- 用户刷新页面后仍能看到历史会话和消息。
- 用户可以切换不同历史会话继续聊天。
- 常见使用场景下，回答质量达到“可用而非完美”的 MVP 标准。
- 系统出现错误时，用户能收到清晰反馈，而不是静默失败。

## 10. 后续迭代方向

- 接入知识库检索与个人资料记忆。
- 增加工具调用，例如联网搜索、日历、邮件、Notion。
- 增加多模态能力，例如图片理解、语音输入输出。
- 增加订阅、配额和计费体系。
- 增加任务型工作流与自动化提醒。
- 增加个性化助手设定与长期记忆。
